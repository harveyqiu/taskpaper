## Project Overview

TaskPaper Lite is a browser-based task manager supporting the TaskPaper format. The repository contains two implementations:

| Version | Entry point | Stack |
|---|---|---|
| **Original** | `index.html` | Vanilla HTML/CSS/JS, no build step |
| **React** (active) | `src/` | React 19 + Vite 6 + TanStack Router + Tailwind CSS v4 |

**To run the React version:**
```bash
npm install
npm run dev      # dev server at http://localhost:5173
npm run build    # production build to dist/
npm test         # unit + component tests (vitest)
npm run test:e2e # E2E tests (requires: npx playwright install)
```

## Architecture

### Three-layer design

1. **React UI layer** (`Toolbar`, `TagColorPicker`) — pure React state and props; safe to re-render freely.
2. **Imperative DOM layer** (`TaskPaperApp` + `Editor`) — the `contenteditable` editor is mounted once by React and then managed entirely via `useRef`. React **never** sets `innerHTML` or `dangerouslySetInnerHTML` on the editor after initial mount. All content mutations, event listeners, and DOM queries use `editorRef.current` directly.
3. **Pure utility layer** (`parser.ts`, `storage.ts`) — plain TypeScript with no React or browser dependencies (safe to unit-test in Node/jsdom).

### State split

`TaskPaperApp` holds two kinds of state:

| Kind | Location | Purpose |
|---|---|---|
| React state (`useState`) | `filter`, `tags`, `tagColors`, `searchQuery`, `matchTotal`, `matchCurrent` | Drives toolbar re-renders |
| Imperative ref (`useRef` → `imp`) | `filter`, `searchQuery`, `tagColors`, `matches`, `matchIndex`, `isComposing`, timers | Read by event handlers without stale-closure issues |

When toolbar state changes (e.g. filter click), React state updates AND `imp.current` is updated synchronously, then the relevant imperative DOM function (`applyFilter`, `renderHighlights`, etc.) is called.

### Data flow

```
User types in editor
  → input event (addEventListener on editorRef)
  → normalizeEditorStructure() → syncEditorClasses()
  → applyFilter(imp.current.filter)          [imperative, no React re-render]
  → scheduleAutoSave()                        [debounced 500ms → saveContent()]
  → setTags(newTags)                          [React state → Toolbar re-renders]
  → scheduleHighlights()                      [rAF → renderHighlights()]

User clicks filter button (React event)
  → handleFilterChange(f)
  → imp.current.filter = f  →  setFilter(f)  →  saveFilter(f)
  → applyFilter(f)                            [imperative]
```

## Key Files

### `src/utils/parser.ts`
Pure functions, no side effects:
- `classifyLine(text)` → `'project' | 'project-done' | 'task' | 'task-done' | 'note' | 'note-done' | 'blank' | 'blank-done'`
- `extractTags(text)` → sorted `['@work', '@today']` (excludes `@done`)
- `extractTagNames(text)` → `['work', 'today']` (used for `data-ptags` attribute)
- `escapeHtml(str)` → escapes `&`, `<`, `>`
- `getIndentLevel(text)` → count of leading tabs or 4-space groups

### `src/utils/storage.ts`
localStorage keys: `taskpaper-lite:content`, `taskpaper-lite:filter`, `taskpaper-lite:tag-colors`.
All functions guard against `typeof window === 'undefined'` (SSR-safe).

### `src/styles/editor.css`
Contains everything Tailwind cannot express:
- CSS custom properties (`--bg`, `--fg`, `--accent`, etc.) with `@media (prefers-color-scheme: dark)` overrides
- `.editor-pane` — the contenteditable div: indent guide via `repeating-linear-gradient` (uses `--tab-px` and `--max-indent-px` set imperatively at runtime), caret color, custom scrollbar
- `.editor-pane > div.line-*` — per-line-type styles (bold for projects, italic for notes, strikethrough for done, `display:none` for `line-hidden`)
- `.hl-rect` / `.hl-current` — search highlight overlay rects

### `src/components/TaskPaperApp.tsx`
The largest file (~850 lines). All imperative editor functions live here as `useCallback`s:
`normalizeEditorStructure`, `syncEditorClasses`, `loadContentIntoEditor`, `applyFilter`, `renderHighlights`, `scrollToMatch`, `navigateMatch`, `toggleDoneOnCurrentLine`, `updateTabWidth`, `updateIndentGuides`, `applyTagColorStyles`, `exportFile`, `importFile`.

A single `useEffect(…, [])` registers all DOM event listeners on mount and cleans them up on unmount.

## TaskPaper Format

- **Projects**: Lines ending with `:` (no leading whitespace) — bold
- **Tasks**: Lines starting with `\t- ` or `    - ` — toggle `@done` with Cmd+D
- **Notes**: Any other indented line — italic, muted color
- **Tags**: `@tagname` anywhere; auto-discovered and rendered as filter buttons
- **Done**: `@done` anywhere in a line marks it as done (strikethrough, muted)

## Keyboard Shortcuts

| Shortcut | Action |
|---|---|
| `Cmd+F` / `Ctrl+F` | Focus search |
| `Cmd+D` / `Ctrl+D` | Toggle `@done` on current line |
| `Tab` / `Shift+Tab` | Indent / unindent (single or multi-line selection) |
| `Cmd+]` / `Ctrl+]` | Indent current line(s) |
| `Cmd+[` / `Ctrl+[` | Unindent current line(s) |
| `Enter` (in search) | Next match |
| `Shift+Enter` (in search) | Previous match |
| `Esc` (in search) | Clear search, refocus editor |

## Implementation Notes

- **contenteditable in React**: `suppressContentEditableWarning` is set on the editor div. React never touches its content after mount — all mutations go through `editorRef.current` to avoid cursor position loss.
- **IME guard**: `isComposing` flag (set on `compositionstart`/`compositionend`) prevents double-processing Chinese/Japanese/Korean input.
- **Search highlights**: Absolutely-positioned `<div>` rects computed from `Range.getClientRects()`, stored in `#highlight-overlay`. Repositioned on scroll and window resize.
- **Tag colors**: Dynamic CSS rules injected into `<style id="tag-color-styles">` in `index.html`'s `<head>` via `applyTagColorStyles()`. Selector uses `data-ptags~="tagname"` attribute.
- **Indent guides**: CSS `repeating-linear-gradient` background on `.editor-pane`. `--tab-px` is measured at runtime by rendering a hidden `<span>` with the editor font; `--max-indent-px` is recomputed after every edit.
- **MutationObserver**: Watches for bare text nodes / `<br>` elements added directly to the editor (can happen on some browser paste/Enter behaviours) and wraps them in `<div>` during `normalizeEditorStructure()`.
